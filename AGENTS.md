## 1. Repository goal & Definition of Done

**Goal**: Sandbox repository to host and exercise an end-to-end Cursor-agent тЖТ CI тЖТ (optional Codex review) тЖТ squash-merge pipeline. Real production code can be added later; the pipeline plumbing must always stay green.

A change is **Done** only when **all** of the following are true:

- [ ] `bash scripts/verify.sh` passes locally (no warnings about skipped steps that should run).
- [ ] The change lives on a feature/fix branch, **never directly on `main`**.
- [ ] A PR exists with **Summary / Test plan / Risk / Related** filled in.
- [ ] CI workflow `build` is green on the PR head commit.
- [ ] Branch protection allowed the merge (means CI was actually checked, not bypassed).
- [ ] Squash-merged into `main`; head branch deleted afterwards.
- [ ] No secrets, no `>50MB` binary, no autogenerated artifacts in the diff.

## 2. The single verification command

**Local and CI run the exact same script:**

```bash
bash scripts/verify.sh
```

If a check needs to be added/removed, **edit `scripts/verify.sh`**. Do not invent a parallel CI-only command; that would let local and CI drift apart (a real-world disaster тАФ see ┬з 8 of the onboarding doc).

If `scripts/verify.sh` fails locally, **fix the code, not the script**. Loosening verify to make a PR pass is a P0 violation.

## 3. Hard rules (violation == stop the agent)

1. **Never `git push` directly to `main`.** Always feature/fix branch + PR + auto-merge.
2. **Never `git push --force` to a referenced remote branch** (any branch that has an open PR or has been pushed before). Force-with-lease to your own private feature branch only after telling the human.
3. **Never bypass `scripts/verify.sh`.** No `--no-verify`, no `[skip ci]`, no commenting out steps.
4. **Never commit anything that looks like a secret.** Tokens, API keys, SSH private keys, `.env` files, OAuth credentials. `.gitignore` already lists them; if `git status` shows a suspicious file, ask before adding.
5. **Never modify another agent's / human's uncommitted work in the worktree.** If `git status` shows files you didn't touch, do not `git add -A`. Stash or ask.
6. **Never modify `~/.ssh/config` existing entries.** Append-only. The owner's machine likely has company GitLab / internal hosts there.
7. **Never modify global `~/.gitconfig` user identity / signing keys** without explicit owner approval.
8. **Existing files (`AGENTS.md`, `.gitignore`, `README.md`, `.cursor/rules/*`) must be appended to, not overwritten.** If a section conflicts, surface the diff to the human first.
9. **One PR = one logical change.** No drive-by refactors. If you find an unrelated bug, open a separate PR.
10. **After every workflow stage, report status in chat and wait for confirmation** before starting the next stage.

## 4. PR rules

- **Title format**: `<type>: <short imperative>` тАФ types are `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`. Examples: `fix: clamp velocity in PD controller`, `chore: enable Codex auto-review`.
- **Body** must contain four sections, in this order:

  ```markdown
  ## Summary
  <1тАУ3 sentences on what changed and why>

  ## Test plan
  - [ ] `bash scripts/verify.sh` passes locally
  - [ ] CI green on head SHA
  - [ ] (any feature-specific manual checks)

  ## Risk
  <What can go wrong? Who/what is affected? Any rollback notes.>

  ## Related
  <Linked issues, prior PRs, upstream references, or "none">
  ```

- **Squash-merge only.** Merge commits and rebase-merges are not allowed (keeps `main` history linear and reviewable).
- **Branch naming**: `<type>/<short-slug>`, e.g. `feat/add-readme-quickstart`, `fix/typo-in-verify`, `chore/enable-codex`.
- **Auto-merge** is enabled at the repo level. The agent should `gh pr merge <num> --squash --auto --delete-branch` immediately after opening the PR; the merge fires only when CI is green and branch protection is satisfied.

## 7. Code review guidelines (P0 / P1 / P2 / P3)

These apply to **both** Cursor agents (when self-reviewing before opening a PR) **and** Codex (when it gets enabled). The point is to focus attention on what actually matters and not waste cycles flagging style nits.

### P0 тАФ BLOCK the merge. Reviewer must request changes.

- Secrets committed (token / private key / `.env` / `*.pem` / DB password).
- `git push --force` to `main` or to a branch with an open PR.
- Control-loop / motor-command / actuator output **without bounds**: missing `clamp` / `min` / `max` on velocity, torque, current, voltage, motor angle, or anything that can physically damage hardware.
- Memory safety: use-after-free, double-free, buffer overflow, missing null check before deref, unchecked `unsafe` block.
- Injection: SQL injection, command injection (`os.system(user_input)`, `subprocess(shell=True, ...)`), template injection.
- `verify.sh` was loosened or skipped (`exit 0` early-return, deleted steps, `|| true` swallowing real errors).
- CI workflow disabled or required check removed from branch protection without explicit owner approval in PR thread.
- License / IP violation (vendored upstream code without preserved license header).

### P1 тАФ Flag and require justification before merge.

- A "magic number" tunable parameter changed by **тЙе30%** (PID gain, learning rate, threshold). The PR body must explain why and what was tested.
- Public API change that is **not backward compatible**: signature change, removed field, renamed exported symbol. Document migration in PR body.
- Tests deleted or commented out without replacement.
- New dependency added: justify the choice (license? maintenance? size?).
- Hot-path performance regression > 2├Ч slower (only if benchmarked).
- New thread / process / async task without clear shutdown story.

### P2 тАФ Optional. Suggest, don't block.

- Refactoring opportunities: duplication, long function, deep nesting.
- Naming clarifications.
- Missing docstring on a non-trivial function.
- Better algorithmic alternative (suggest, link reference).

### P3 тАФ Do not flag. These waste reviewer attention.

- Typos in comments.
- Style / formatting (defer to formatter тАФ `black`, `clang-format`, `prettier`).
- "Use `f-string` instead of `.format()`" sort of preferences.
- Subjective opinions ("I would have written it differently").

When Codex writes a textual review, it must classify each comment as P0/P1/P2/P3 in the message. Comments without a classification can be ignored by the human.

## 8. What to do when something goes wrong

| Symptom | First action | Where to look |
|---|---|---|
| `bash scripts/verify.sh` fails locally | Read the failing step output. Do **not** edit `verify.sh` to silence it. | `scripts/verify.sh` step number in the output |
| CI red after PR opened | Click "Details" on the failed check on the PR page. | `.github/workflows/ci.yml` |
| PR auto-merged immediately without waiting for CI | Branch protection misconfigured (contexts is empty set). Re-do ┬з 5 of onboarding doc, ensure `build` is in required checks list. | `gh api repos/xidikr/TEST/branches/main/protection` |
| Cannot push (`Permission denied (publickey)`) | SSH key issue. Verify `ssh -T git@github.com` returns `Hi xidikr!`. If not, re-check `~/.ssh/config` and `~/.ssh/id_ed25519_github`. | ┬з 1 / ┬з 2 of onboarding doc |
| `gh pr create` fails with `No commits between...` | GraphQL indexing race. Wait 5s, or use REST API: `gh api repos/xidikr/TEST/pulls -X POST ...`. | Onboarding doc ┬з 8 хЭС #2 |
